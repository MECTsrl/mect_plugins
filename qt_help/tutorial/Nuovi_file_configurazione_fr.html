Nouveaux fichiers de configuration dans les projets avec la version 3.4.x

Avec la version 3.4.x, deux nouveaux fichiers ont été ajoutés pour les nouveaux projets, hmi.ini et hmi.qss, regroupés dans la nouvelle section “Other files”.


File hmi.ini:

Le fichier hmi.ini est un fichier texte au “Windows Ini Format”, organisé en Sections (titres entourés de []) et Clé = Valeur paires

		[General] 
		rotation=0 
		plc_host=127.0.0.1
		disable_all_nodes=false

Le fichier hmi.ini est automatiquement copié dans le même dossier de l'application lors du déploiement. Son chemin cible est /local/root/hmi.ini.

Au démarrage, le programme hmi lira la clé “disable_all_nodes” dans la section [General], et si la valeur est définie sur true, les nœuds distants définis dans l'application ne seront pas automatiquement activés.
La valeur par défaut est false pour ne pas modifier le comportement des applications existantes.

Il est donc désormais possible de définir des configurations complexes dans Crosstable et d'activer dans le code cpp, selon la configuration sur le terminal, uniquement les nœuds distants nécessaires, avec l'instruction:
doWrite_NODE_xx_STATUS(1);

Le fichier hmi.ini peut être utilisé pour stocker d'autres paramètres d'application, de préférence en utilisant des sections autres que [General], par exemple:

		[mySession] 
		myKey=myValue

Pour lire la valeur de la clé myKey dans la section mySession il suffit, dans le code cpp:

#include<QSettings>

		QSettings hmi_ini("/local/root/hmi.ini", QSettings::IniFormat);
		QString myValue = hmi_ini.value("mySession/myKey").toString()

File hmi.qss:

Le fichier hmi.qss est un fichier texte au format “Qt Style Sheet” (similaire à HTML Cascading Style Sheet CSS).

Le fichier hmi.qss est automatiquement copié dans le même dossier de l'application lors du déploiement. Son chemin cible est /local/root/hmi.qss.

Au démarrage, le programme hmi lira (s'il existe) le fichier “hmi.qss” dans la fonction main et interprétera les directives qu'il contient à tous les objets de l'application, en exécutant ce code (dans la bibliothèque):

		// Loading Application QSS 
		QFile fileQSS(“/local/root/hmi.qss”);
		if (fileQSS.exists())  {
		fileQSS.open(QFile::ReadOnly);
		QString styleSheet = QString(fileQSS.readAll());
		fileQSS.close();
		app.setStyleSheet(styleSheet);
		qDebug(“Loaded hmi.qss”);
		}

Pour être efficace, cette opération doit être effectuée avant la création des objets graphiques de l'application (pages utilisateur et système).

Si le fichier n'existe pas, est vide ou a un contenu invalide, il sera ignoré.

Le fichier fourni dans les nouveaux projets est assez complexe, car il est conçu pour modifier uniquement les pages de la bibliothèque et n'a aucun effet sur l'application développée par l'utilisateur, définissant une sorte de “Dark Theme” pour les pages de la bibliothèque.

L'utilisateur a la possibilité d'ajouter de nouvelles directives QSS pour ses pages ou de modifier celles existantes pour normaliser l'apparence finale du projet.

Pour plus d'informations, veuillez consulter: https://doc.qt.io/archives/qt-4.8/stylesheet-examples.html#style-sheet-usage


Passez au fichier automation.ccp:

Pour les nouvelles applications, le fichier automation.cpp a été modifié:

En raison de l'optimisation de la phase de démarrage des appareils et des modifications du protocole de communication interne entre l'automate et l'hmi, introduites avec la version 3.3.8 de Mect Suite, une boucle d'attente a été introduite dans la fonction setup() de automation.cpp que la variable système PLC_EngineStatus atteint la valeur 2 (enRunning)

		/* put here the initalization */ 
		void setup(void)
		{    
			// Wait PLC Engine gets ready
			while (PLC_EngineStatus < 2) {
				fputc('*', stderr);
				sleep(1);
			}
			// Insert your start-up code here
			// .....
		}

Avec des valeurs inférieures (enIdle = 0, enInitialized), la communication entre l'IHM et l'automate n'est pas garantie et il n'est pas possible de lire ou de définir des valeurs fiables pour les variables Crosstable.

Par conséquent, les décisions basées sur les valeurs des variables Crosstable ou les paramètres de nouvelles valeurs doivent intervenir après cette boucle d'attente.
